This manual describes the features and customization options for the Emacs Lisp package =agent.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=agent.el= collects the Emacs integration layer for AI coding command-line tools. It provides a shared session model, a unified transient menu, notification handling, terminal helpers, backend modules for Claude Code and Codex, and an optional chief-of-staff loop.

The package is split into a small core and backend modules. The core file, =agent.el=, is backend-agnostic: it knows how to register AI tools, find their buffers, dispatch shared commands, render backend icons, assign stable session keys, and present a single menu. =agent-claude.el= integrates =claude-code.el= with Claude-specific account switching, status polling, usage polling, branch navigation, batch TODO processing, and hook setup. =agent-codex.el= integrates =codex.el= with Codex-specific theme sync, skill execution, handoff, auditing, debugging, and modeline helpers. =agent-chief.el= implements an inspectable chief-of-staff loop: Emacs owns timers, state, context collection, and notifications, while Claude Code or Codex provides one-shot judgment on each tick.

The main workflow is to load =agent.el=, then require the backend modules you use. Once the Claude and Codex modules are loaded, =M-x agent-menu= and =M-x agent-start-or-switch= operate across both tools. Sessions are grouped by account when the backend exposes account metadata, highlighted when waiting for input, and disambiguated by project and instance name when several sessions share a directory.

The package depends on =transient= for menus and =consult= for completion. The Claude backend expects =claude-code.el=. The Codex backend expects =codex.el=. Optional integrations use packages such as =yasnippet=, =doom-modeline=, =gptel=, =agent-log=, and =monet= when those are available.

The user option ~agent-protect-buffers~ is a boolean that defaults to ~t~. When enabled, killing a live AI session buffer prompts for confirmation through ~agent-protect-buffer~. Buffers whose processes have already exited are allowed to die without the prompt.

The user option ~agent-alert-on-ready~ is a boolean that defaults to ~nil~. When enabled, backend notification handlers can call ~agent-notify~ to emit visual and/or audible alerts when a session is waiting for input. Codex's own terminal-bell notification path is routed through the same alert setting. ~agent-alert-style~ selects ~visual~, ~sound~, or ~both~; it defaults to ~both~. ~agent-alert-sound~ defaults to ~nil~; set it to a sound file to enable sound alerts. ~agent-alert-sound-player~ optionally names an external sound player when Emacs sound support is unavailable.

The user option ~agent-backtrace-file~ controls where ~agent-save-backtrace~, ~agent-claude-debug-backtrace~, and ~agent-codex-debug-backtrace~ write Emacs backtraces. It defaults to a file in ~temporary-file-directory~.

The user options ~agent-epoch-project-registry-file~ and ~agent-epoch-projects-root~ configure Slack-message project routing. The default registry is the generated =automations-dashboard/repo/data/automations.json= file, which is consumed by the Epoch automations dashboard and points back to the canonical project Org metadata.

The user option ~agent-prompt-capture-directory~ controls where ~agent-capture-prompt~ stores per-session Org files. Each file is keyed by backend, account, project directory, and session instance, so prompts are linked to the session they target and remain available after Emacs restarts. ~agent-prompt-capture-auto-save-delay~ controls the idle delay before capture buffers are saved automatically.

The user option ~agent-sync-theme~ is a boolean that defaults to ~nil~. When enabled, theme changes in Emacs are persisted through each registered backend's theme-sync handler. This updates configuration files for new or restarted CLI sessions without sending theme commands into active terminal conversations.

The user option ~agent-sigwinch-delay~ is a number of seconds and defaults to =0.5=. It controls how long ~agent-fix-rendering~ waits before sending =SIGWINCH= to a terminal process. Increase it if an AI CLI reads the terminal size before the =eat= window has settled.

Snippet helpers use =yasnippet= and =consult-yasnippet= only when those optional packages are loaded.

The Claude backend options control account selection, session metadata, non-interactive execution, and branch workflows. ~agent-claude-accounts~ is an alist of account names to Claude configuration directories, and ~agent-claude-account-file~ stores the selected account across Emacs sessions. Use ~agent-claude-select-account~ or the account infix in ~agent-menu~ to switch accounts before starting a new session.

Status and usage polling are controlled by integer intervals. ~agent-claude-status-interval~ defaults to =5= seconds and reads status JSON emitted by the bundled statusline script. ~agent-claude-usage-interval~ defaults to =300= seconds and backs off up to ~agent-claude-usage-max-interval~, which defaults to =900= seconds.

Batch execution options are passed to =claude -p=. ~agent-claude-batch-permission-mode~ defaults to ="auto"=, ~agent-claude-batch-max-turns~ defaults to =30=, and the allowed tool, model, and appended-system-prompt options let you narrow or expand what non-interactive Claude tasks may do.

Branch and buffer safety options default conservatively. ~agent-claude-warn-kill-with-branches~ defaults to ~t~ and warns before killing a session that belongs to a branch tree. ~agent-claude-fork-worktree-directory~ defaults to a cache directory so isolated branch worktrees do not interfere with cloud sync. ~agent-claude-handoff-file~ defaults to a file in ~temporary-file-directory~.

Claude hook setup is explicit. ~agent-claude-settings-file~ names the settings file updated by ~agent-claude-setup-config~ and the individual ensure commands. ~agent-claude-hook-wrapper~ points to the upstream =claude-code= hook wrapper, while ~agent-claude-statusline-script~ points to the bundled statusline helper. ~agent-claude-status-directory~ controls where the helper writes per-buffer status JSON files. The statusline setup refreshes an existing agent-owned =claude-code-statusline.sh= command, which keeps the settings file correct after the package moves or is renamed, while preserving unrelated custom Claude statusline commands. Loading the backend does not modify Claude Code configuration.

The Codex backend options control account selection, handoff, skill discovery, non-interactive execution, project auditing, and backtrace debugging. ~agent-codex-accounts~ is an alist of account names to Codex home directories, and ~agent-codex-account-file~ stores the selected account across Emacs sessions. Account homes keep credentials such as =auth.json= separate, while shared Codex state such as =config.toml=, =hooks.json=, skills, sessions, and =history.jsonl= is symlinked from =~/.codex/= so hooks, skills, project trust, and chat history remain available after switching accounts. Use ~agent-codex-select-account~ or the Codex account infix in ~agent-menu~ to switch accounts before starting a new session. ~agent-codex-handoff-file~ defaults to =/tmp/codex-handoff.md= and is read by ~agent-codex-handoff~ after the Codex handoff skill writes next-session instructions.

~agent-codex-skill-directories~ is a list of additional directories scanned for Codex skills. The backend also scans standard Codex skill locations and project-local skill directories. ~agent-codex-audit-skills~ defaults to the code, design, and interpretability audit skills, while ~agent-codex-audit-project-directories~ stores completion candidates for the audit directory prompt.

~agent-codex-exec-approval-policy~, ~agent-codex-exec-sandbox-mode~, and ~agent-codex-exec-skip-git-repo-check~ configure the non-interactive Codex calls used by skill and audit workflows. The approval policy is passed as a top-level Codex CLI option before the =exec= subcommand, while the sandbox and repository-check options are passed to =codex exec= itself.

~agent-codex-debug-backtrace-model~ and ~agent-codex-debug-backtrace-backend~ configure the =gptel= request used by ~agent-codex-debug-backtrace~ to identify likely culprit packages from an Emacs backtrace.

The Slack-message routing options, ~agent-claude-act-on-slack-message-model~, ~agent-claude-act-on-slack-message-backend~, ~agent-codex-act-on-slack-message-model~, and ~agent-codex-act-on-slack-message-backend~, configure the =gptel= request used to select an Epoch project from the Slack message at point. The selected backend starts in that project's canonical directory from the project document path, not a nested repo, and inserts the Slack URL into the prompt without submitting it, so the prompt can be reviewed first.

The optional =agent-chief= module runs a periodic, auditable chief-of-staff workflow. Emacs owns the schedule, explicit state, and notifications. In the default =session= mode, the selected backend is an ordinary long-lived Claude Code or Codex session that receives heartbeat prompts and remains available for normal conversation.

~agent-chief-backend~ selects =codex= or =claude-code=. ~agent-chief-mode~ defaults to =session=; set it to =stateless= to use the older one-shot JSON-decision path. ~agent-chief-interval~ controls the timer interval in seconds. ~agent-chief-directory~ is the working directory for chief sessions and one-shot calls; by default it uses =/Users/pablostafforini/My Drive/Epoch/= when that directory exists, falling back to ~user-emacs-directory~ otherwise. ~agent-chief-session-instance-name~ defaults to =chief= and is used as the instance name for the dedicated interactive session.

~agent-chief-state-file~ is an Org file used as the durable, inspectable state store; =agent-chief= never writes opaque memory. Day plans and manual notes are appended there and, when a chief session is live, submitted into that session as conversational context.

State context treats only a =Plan YYYY-MM-DD= heading dated today as current. Older plan headings are supplied as historical context, not active obligations, unless restated in today's plan.

~agent-chief-record-model-state-updates~ controls whether model-proposed state updates from stateless ticks are appended to the state file. It defaults to nil because periodic model observations are not user-approved durable memory. Use explicit notes and day plans for ordinary state updates.

~agent-chief-context-functions~ is the extension point for agenda, task, calendar, project, or custom status snippets. Each function is called with no arguments and should return a string. Nil and empty strings are ignored. The state file is included separately, truncated to ~agent-chief-state-max-chars~ characters. ~agent-chief-max-turns~ limits Claude Code ticks; Codex ticks use the existing =codex exec= settings from =agent-codex=.

In session mode, ~agent-chief-start-session~ sends only a short conversational kickoff asking the model to ask for today's plan; it does not dump the state file or stateless JSON contract into the visible transcript. Heartbeat prompts ask the model to respond exactly =No nudge.= when no contact is warranted, or =Nudge: MESSAGE= when it should alert the user. The backend's ordinary notification event tells =agent-chief= when the response is complete; =agent-chief= then inspects the session text since the heartbeat and calls ~agent-chief-notify-function~ only for a nudge. The default notification function logs the message and uses =alert.el= when available. ~agent-chief-system-prompt~ is the standing instruction that makes the loop selective and forbids inventing obligations not present in the supplied context.

In stateless mode, backend output may include echoed CLI transcript text before the answer. The chief loop scans that output and uses the last valid JSON object with a =notify= field as the decision, so echoed response-schema examples do not confuse the parser.

The command ~agent-menu~ opens the unified transient menu. It includes shared entries for session switching, waiting-session jumps, handoff, restart, exit, skill execution, project audit, backtrace debugging, Slack-message action routing, alert toggles, terminal maintenance, and a separate Prompts section for prompt capture and captured prompt insertion. Backend modules append their own suffixes to the same menu when loaded.

The command ~agent-start-or-switch~ is the main entry point. If no sessions exist, it starts a new session through the selected backend. If sessions already exist, it opens the session switcher. Sessions receive stable home-row keys where possible and display backend icons next to their labels.

The command ~agent-run-skill~ discovers skills from every loaded backend that implements skill discovery. It disambiguates same-named skills by backend label, prompts for declared arguments when a skill frontmatter requests them, and dispatches the invocation to the selected backend. Claude and Codex backends scan global and project-local programmatic skill directories, using both ~project-current~ and nearby tool directories as project-root signals, so batch-only skills can be run from this command without being loaded into ordinary interactive agent sessions.

The command ~agent-capture-prompt~ opens a session-specific Org file and appends a new prompt entry. The command ~agent-insert-captured-prompt~ reads pending nonempty entries for the current session, prompts for one with a short body preview in the completion candidate, inserts its text into the backend prompt without submitting it, and marks the Org entry with an =INSERTED= timestamp. With a prefix argument, it includes already inserted entries.

The command ~agent-post-push-ci~ runs the ~post-push-ci~ skill through the
selected backend with =--no-push --commit HEAD=. Use it after a push has
already happened and you want an agent to inspect GitHub Actions failures, fix
the root cause, and continue the authorized CI closeout loop.

The commands ~agent-handoff~, ~agent-audit-project~, ~agent-debug-backtrace~, ~agent-act-on-slack-message~, ~agent-exit~, and ~agent-restart~ resolve a backend from the current session buffer or by prompting. This lets the same keybinding operate on Claude or Codex sessions without separate wrapper commands.

~agent-before-exit-functions~ can delay ~agent-exit~ for cleanup work.
Each function receives the resolved backend and session buffer; returning nil
aborts the exit.

Set ~agent-before-exit-skill-name~ and
~agent-before-exit-skill-directories~ to submit an interactive skill before
exiting matching sessions. ~agent-run-skill-before-exit~ uses
~agent-skill-command-prefix-alist~ so Claude sessions receive slash commands
and Codex sessions receive dollar-prefixed skill commands, then submits the
command. When the backend reports that the skill run has finished, the
session exits automatically; ordinary ready notifications are left unchanged.
Leave ~agent-before-exit-skill-directories~ nil to run the skill before
exiting every session. ~agent-before-exit-skill-min-duration-seconds~ defaults
to 60 seconds, so very short sessions skip the before-exit skill when the
backend can report elapsed session time.
Set the buffer-local variable ~agent-before-exit-skill-inhibit~ non-nil for
orchestration sessions that must close immediately without running the
configured before-exit skill. Handoff commands set this automatically on the
source session before closing it.

Backends can veto that automatic close while the submitted skill invocation is
still visible as unaccepted prompt input. Codex uses this to avoid closing a
session when the dollar-prefixed skill name was inserted but not submitted,
while ignoring submitted command scrollback, visible placeholder text, and
autosuggestion text.

Backends can provide an atomic submit operation for before-exit commands.
Claude and Codex both target the resolved session buffer directly, insert the
skill command, and submit it once before the terminal buffer is displayed
again. Codex delegates this to =codex.el='s target-buffer submit helper.

The command ~agent-chief-start-session~ starts or reuses a dedicated Claude Code or Codex session using ~agent-chief-session-instance-name~, sends the chief-of-staff operating contract into the session, asks the model to request today's plan there, and starts the heartbeat timer. The command ~agent-chief-switch-to-session~ jumps back to that session so the user can reply normally.

The command ~agent-chief-start~ starts only the repeating Emacs timer. Without a prefix argument it also schedules an immediate tick; with a prefix argument, it waits ~agent-chief-interval~ seconds before the first tick. ~agent-chief-stop~ cancels the timer. ~agent-chief-tick~ runs one check using ~agent-chief-mode~. ~agent-chief-session-heartbeat~ forces a session-mode heartbeat, while ~agent-chief-stateless-tick~ forces one stateless JSON-decision check.

The command ~agent-chief-open-state~ opens the auditable Org state file. ~agent-chief-add-note~ appends a manual note, and ~agent-chief-set-day-plan~ appends today's plan. When a chief session is active, both commands also submit the new context into that session. These commands are intentionally simple: the durable source of truth remains ordinary text that can be reviewed, edited, and audited outside the model.

The function ~agent-register-backend~ registers an AI tool by associating a backend symbol with a property list of buffer predicates, session discovery functions, command senders, and optional command handlers. Backend modules call it at load time. Required keys are validated so the shared session layer can rely on a consistent interface.

~agent-display-name~ returns a human-readable label for a session buffer. It starts with the project name, adds an instance name when needed to disambiguate duplicates, and lets backends append their own suffixes. The Claude backend uses this to expose branch identity in the modeline and switcher.

~agent-parse-skill-frontmatter~ reads =SKILL.md= frontmatter shared by Claude and Codex skill runners. It recognizes names, descriptions, argument hints, argument sources, argument choices, defaults, multiplicity, user-invocable flags, and model overrides.

~agent-svg-icon~ renders SVG strings as inline images, using the current face foreground when the SVG uses =currentColor=. Backends use this for modeline and transient display and can fall back to text abbreviations when SVG rendering is unavailable.

The Claude and Codex status helpers are intended for modeline integrations such as =doom-modeline=. Claude status functions read the current session status JSON and expose model, cost, context, duration, and token information. Claude usage functions expose account-level utilization returned by the Anthropic usage endpoint. Codex status functions expose the configured model and elapsed session duration. The session switcher uses backend busy and background-work detectors to avoid showing actively responding sessions as waiting, and to distinguish truly idle waiting sessions from sessions that are waiting for input while background terminals, agents, or tasks are still running.

* Transient menu
:PROPERTIES:
:CUSTOM_ID: h:transient-menu
:END:

#+findex: agent-menu
Backend modules extend ~agent-menu~ through ~transient-append-suffix~ after the core menu is available, first removing their own existing suffixes so reloading does not duplicate or stale-position those entries. The base menu contains shared session, tool, alert, buffer, prompt, and option entries. The Claude module appends branch navigation, batch TODOs, TODO sending, log browsing, status polling, Claude account selection, and branch-kill warnings. The Codex module appends Codex resume/fork commands and Codex account selection. In the Options group, =-c= selects the Claude account and =-x= selects the Codex account so the two account selectors appear next to each other.

Boolean menu infixes use a shared transient class that toggles the bound variable each time it is invoked. The theme-sync infix additionally calls ~agent-sync-theme-now~ when enabling theme sync, so backend configuration files are updated immediately.
